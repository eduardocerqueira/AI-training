Q: What is this project?
A: A small FastAPI service that answers questions using a local Q&A text file and Hugging Face Inference Providers. A Gradio UI calls the API for local testing.

Q: How does answering work (v1)?
A: The full knowledge/qa.txt file is injected into the system prompt. There is no vector database or embeddings yet — the model reads the entire Q&A file each request.

Q: What Python version is required?
A: Python 3.11 or newer.

Q: How do I set up the project?
A: From apps/python/hf-qa-rag: create a venv, run pip install -e ".[dev,ui]", copy .env.example to .env, and set HF_TOKEN from https://huggingface.co/settings/tokens.

Q: How do I run the API?
A: From apps/python/hf-qa-rag with the venv active: hf-qa-rag-api, or uvicorn hf_qa_rag.api:app --reload --port 8000. API docs are at http://127.0.0.1:8000/docs.

Q: How do I run the chat UI?
A: Start the API first, then run hf-qa-rag-ui (requires pip install -e ".[ui]"). Open http://127.0.0.1:7860. The UI reads API_URL from .env (default http://127.0.0.1:8000).

Q: What if port 8000 is already in use?
A: Another process is bound to 8000. Stop it (e.g. lsof -i :8000) or start uvicorn on another port: uvicorn hf_qa_rag.api:app --reload --port 8001, and set API_URL to match for Gradio.

Q: Where is the knowledge base stored?
A: In knowledge/qa.txt under the app folder (apps/python/hf-qa-rag/knowledge/qa.txt). Override the path with KNOWLEDGE_PATH in .env.

Q: How do I update the knowledge base without restarting?
A: Edit knowledge/qa.txt, then POST /knowledge/reload, or restart the API. GET /knowledge shows path, character count, and a preview.

Q: What environment variables are supported?
A: HF_TOKEN (required for /chat), HF_MODEL (Hub model id for chat_completion), KNOWLEDGE_PATH (default knowledge/qa.txt), API_URL (Gradio only, default http://127.0.0.1:8000).

Q: How do I choose a Hugging Face model?
A: Set HF_MODEL to a Hub model id (e.g. meta-llama/Llama-3.1-8B-Instruct). On huggingface.co, open the model page and check Inference Providers — at least one provider must be enabled in your account at https://huggingface.co/settings/inference-providers.

Q: Why do I get model_not_supported or a 502 from /chat?
A: The model is not hosted on any Inference Provider you have enabled. Enable a matching provider in HF settings, or pick a model that lists your providers (e.g. Groq, Novita, Cerebras, Featherless) on the model page. Filter models at https://huggingface.co/models?inference=warm&pipeline_tag=text-generation.

Q: Does hf auth login replace HF_TOKEN in .env?
A: No. This app reads HF_TOKEN from .env via pydantic-settings. CLI login is optional for other tools.

Q: What API endpoints exist?
A: GET /health (status and model), GET /knowledge (KB metadata and preview), POST /knowledge/reload (reload qa.txt), POST /chat (body: message and optional history).

Q: What is the /chat request format?
A: JSON with "message" (required string) and optional "history": list of {"role": "user"|"assistant", "content": "..."}. Only the last 10 history turns are sent to the model.

Q: What does /chat return?
A: JSON with "answer" (string) and "sources" (list of paths, usually the knowledge file relative to the app root).

Q: How do I test /chat from the terminal?
A: curl -s http://127.0.0.1:8000/chat -H 'Content-Type: application/json' -d '{"message": "How do I run the API?"}'

Q: How do I run tests?
A: From apps/python/hf-qa-rag with dev dependencies installed: pytest.

Q: What happens if HF_TOKEN is missing?
A: /chat returns 503 with a message to copy .env.example to .env and set HF_TOKEN.

Q: Are there gated models (e.g. Llama)?
A: Some Hub models require accepting the license on the model page while logged in with the same account as your token, before inference works.

Q: What are the limitations of v1?
A: Entire knowledge file in one prompt (size limited by model context), no chunking, no semantic search, no citation of specific Q&A pairs — only the file path in sources.

Q: What are suggested next steps for production?
A: Replace qa.txt with real domain Q&A, deploy the API, point Gradio or a frontend at the public URL, and add chunking plus embeddings when the knowledge base outgrows the context window.
